1. Quick start — your first valid packet in 5 minutes

If you want to understand the system deeply, read everything below. If you want to see it work first, do this:

1
Tab 1 — Scenario. Open the Scenario Preset dropdown and select 50 kW Baseline — Cold Loop + Hot Backbone (GEO). The system seeds a complete coherent state. You're already 80% done.
2
Tab 7 — Output & Packet. Scroll to the bottom and click Build Report. The runtime executes and returns authoritative numbers: thermal load, radiator area, risk summary, topology validation.
3
Read the output. Look at Thermal Load Summary, Radiator Sizing, and the Risk Summary card. The green runtime ✓ badges confirm server-side execution — these are real outputs, not estimates.
4
Click Open Formatted Report for a clean printable version of everything. Download the JSON bundle to keep the packet artifact.

That's a valid run. Now go back to any tab and change things. The system is designed to be explored. Change the device count, swap the radiator material, add a branch — then run again and compare. That iteration is the whole point.

The rest of this guide explains what you're actually changing and why it matters.

2. The core mental model

One sentence that explains everything:

The browser is a packet builder. The server is the only calculator.

The UI lets you assemble, inspect, and edit a scenario. The moment you click Build Report, the entire packet goes to a server-side runtime engine. The runtime normalizes, validates, aggregates, and computes. The numbers it returns are the only numbers you should trust for engineering decisions.

Preview values shown while you edit — like the radiator area preview in Tab 5 or the compute total preview below Tab 2 — are convenience indicators. They help you steer. They are not the output.

Three kinds of information exist in the UI:

  • Runtime Server-executed. These numbers matter.
  • Preview Browser-calculated estimates. Useful for steering, not authoritative.
  • Exploratory Intentionally non-authoritative. Used for concept comparison, not engineering claims.

The seven tabs move in a physical order: mission frame → heat source → other loads → transport topology → rejection and storage → optional side loops → authoritative output. Follow that order the first time through. After that, edit in any order you like.

4. Tab 1 — Scenario

Tab 1 is the control frame. Everything downstream inherits from it.

The fields that matter most

Architecture Class is the most important structural selector in the entire UI. It declares what kind of thermal plant you are about to describe in Tabs 4 and 5. If you select cold_loop_plus_dual_hot_island here, Tab 4 should have zones that match that topology or the runtime will raise flags.

Mission Mode is a major behavioral switch. full_compute means continuous full load — the hardest thermal case. burst means intermittent peak load with cooldown intervals. eclipse changes what storage strategies are sensible. safe_mode drops most load. Pick the mode that matches your trade scenario, not just the most demanding case every time.

Assumptions field: use this. One line per assumption. Examples: "assumed radiator symmetric deployment", "assumed NaK compatible with SS-316 at operating temps", "assumed burst duty cycle 20%". The runtime cannot infer these from numbers alone. If you don't surface them here they become invisible bugs in your trade analysis.

Extension toggles

The four extension toggles control which modeling layers are active. You do not need them all on for a valid basic run.

5. Tab 2 — Compute Payload

Tab 2 defines the primary heat source — the compute module — and everything about how it generates and tolerates heat.

Device count and load state

Device count scales the array. Target load state picks which rung of the power ladder the runtime uses. A system with 8× H200-class GPUs at full load is a 5600 W compute electrical draw before overheads. At medium load that becomes roughly 3500 W. The radiator burden tracks this directly.

Module overheads — don't leave these at zero

Real compute modules have memory, storage, network fabric, power conversion hardware, and control systems that all generate heat. These are separate overhead fields for exactly that reason. If you leave them at zero you are modeling a GPU floating in space with no support electronics, which is not a real scenario. Add realistic overhead estimates. Even modest overheads can add 1–2 kW per module.

Redundancy mode

This changes installed vs active capacity. n+1 means one spare — installed count is device_count+1 but thermal modeling reflects active fraction. This matters for sizing because you may need to handle the worst-case active set, not just the nominal set.

6. Tab 3 — Non-Compute Payload

This tab is often skipped on first pass and it shouldn't be. A scenario with only compute load and zero payload overhead is not a spacecraft scenario — it's a heater-in-a-vacuum scenario. Real missions have TT&C, crosslinks, optional radar or optical comms, and power management loads. These add to the radiator burden and they are not in the scenario presets.

Why additive blocks

Payload service loads are not one number. A satellite may run TT&C continuously, fire its crosslink at 50% duty cycle, and activate its radar only during imaging passes. Additive blocks let you model that structure instead of collapsing everything into one field.

Duty modes

continuous means the load is always on. uniform applies a duty fraction across all subsystems equally. per_subsystem lets each load item have its own duty cycle. Use per_subsystem when your payload has genuinely different operating profiles for different loads.

7. Tab 4 — Thermal Architecture

This is where the system becomes an actual thermal network instead of just a list of watts. Tab 4 is the hardest tab to understand and the most important one to get right.

Zones — what they are

A thermal zone is not just a temperature point. It is a named location or function in the thermal network that carries physical meaning: where heat is picked up, transported, exchanged, stored, or rejected.

Every zone declares: what role it plays, what type of object it is, what chain it belongs to, what its upstream and downstream neighbors are, and what materials and working fluids are involved.

Zone roles vs zone types

Zone role answers: what job does this zone perform? (pickup, transport, rejection, storage, convergence exchange)
Zone type answers: what thermal object is this? (compute_vault, hx_boundary, hot_backbone, hot_island, storage_buffer, radiator_zone)

These can seem redundant. They are not. A zone might have role pickup and type compute_vault — that means it is a pickup zone located in the compute vault. Another might have role convergence_exchange and type hx_boundary — meaning it is the HX interface where two loops meet.

The upstream/downstream graph — most important thing in this tab

Upstream and downstream refs turn your list of zones into a directed graph. Without them, zones are just labeled boxes. With them, the runtime can validate topology, detect cycles, confirm chain continuity, and order execution.

Think of it physically: heat flows from the compute vault → along the cold loop → across the HX boundary into the hot backbone → to the radiator zone. Each arrow in that sentence is a downstream ref declaration.

HX touch points — the controlled interface concept

A heat exchanger boundary is where two distinct thermal loops or domains are allowed to interact without becoming the same loop. This is one of the most important concepts in multi-loop architecture modeling.

Real architectures use HX boundaries to:

  • Keep compute electronics at a cool operating temperature while running the rejection loop at a much higher temperature
  • Allow two hot islands to exchange heat without being wired together
  • Let a Brayton or Stirling cycle draw from a hot-side source without the compute loop seeing those temperatures

The rule: if two thermal paths need to share energy but should not share temperature directly, use an HX boundary zone between them. Declare convergence enabled on that zone if the runtime should iterate to equilibrium.

Resistance chain fields

The R-value fields (junction→case, case→spreader, spreader→pickup, pickup→loop, loop→sink) describe the local thermal resistance ladder through the device package and into the coolant. Null values tell the runtime to use defaults. Non-null values let you model a specific package-to-loop resistance path for more detailed lifecycle or temperature budget analysis.

BOL/EOL lifecycle fields (Extension 3A)

BOL is Beginning of Life. EOL is End of Life. In space, radiator emissivity degrades over time due to radiation damage and contamination. The system models this as a degradation fraction applied to the BOL emissivity to derive EOL emissivity. A radiator that starts at ε=0.90 and degrades 5% arrives at ε=0.855 at EOL — requiring larger area to maintain the same rejection.

The display shows both BOL and EOL area estimates side by side so you can see the margin you need to design for over mission life.

9. Tab 6 — Optional Branches

Branches are side-path systems that attach to the main thermal architecture and either extract energy, convert it, or export it. None are required for a valid packet. All add architectural complexity that must be physically coherent.

Before enabling any branch: make sure your Tab 4 zone graph has a declared source zone that the branch can draw from. A Brayton branch needs a hot-side source zone. A TPV branch needs a high-temperature emitter zone. A branch with no coherent thermal source will either raise a blocker or produce physically meaningless output.

⚡ Reverse Brayton / Heat-Lift Branch

A reverse Brayton cycle uses work input (electrical power) to move heat from a cold source to a hotter rejection domain — the thermal equivalent of a heat pump or refrigerator. This is useful when you have a compute zone that cannot tolerate the hot-side rejection temperature and you need to actively lift that heat to a higher domain before rejecting it. The tradeoff: the electrical work consumed to run the compressor becomes additional heat the system must ultimately reject, increasing the total radiator burden.

When it helps: when a zone is thermally isolated from the hot backbone by a large temperature gap that passive transport cannot bridge.

🔄 Brayton Power-Cycle Branch

A true Brayton power cycle extracts work from a high-temperature thermal source. Working fluid (typically He/Xe gas mixture in space applications) is compressed, heated by a high-temperature source, expanded through a turbine to generate electrical work, then rejected at a lower temperature. This is not a heat pump — it is an engine. It requires a genuinely hot thermal source (typically above 1000 K to be meaningful) and it produces electricity that can offset solar array requirements or power onboard systems.

The catch: the waste heat from the cycle's cold-side rejection still requires a radiator. You get electrical output but you do not eliminate rejection.

When it helps: when you have a high-grade thermal source from compute waste or solar concentration and you want to reclaim electrical work from it rather than just radiating it away.

🔁 Stirling Branch

A Stirling engine operates on a closed regenerative thermodynamic cycle using a working gas (hydrogen or helium typically) reciprocating between hot and cold spaces. Unlike Brayton, Stirling is a piston-based cycle — mechanically simpler at small scale, but lower power density. Stirling engines have been used on spacecraft for cryocooling (reverse Stirling) and have been studied for direct conversion of radioisotope or solar thermal heat to electricity. In the context of this system, a Stirling branch draws from a hot-side thermal domain and generates electrical work.

When it helps: moderate-temperature power recovery at smaller scale than a full gas-turbine Brayton cycle.

📡 TPV — Thermophotovoltaic Branch

TPV converts infrared radiation from a hot emitter surface directly to electricity via photovoltaic cells — no moving parts, no working fluid. The hot emitter radiates photons; PV cells tuned to that wavelength absorb them and generate current. Efficiency is highly sensitive to emitter temperature. At 800–1000 K, practical TPV efficiency is poor (sub-10%). At 1200–1600 K with well-matched PV bandgaps, efficiency can reach 20–30% in research settings.

The constraint: TPV PV cells need to be cooled. Their waste heat still needs a radiator behind them. TPV reduces the load on the main radiator only if the electrical output is exported off-vehicle or used to power systems that would otherwise require solar array area. If the electricity is consumed onboard, it becomes heat again.

Extension 4 in this system models TPV as an exploratory recapture layer — intentionally labeled non-authoritative because TPV efficiency coefficients at space-relevant temperatures remain research-stage.

When it helps: when you already have a high-temperature emitter surface (above 1000 K) and want to quantify what electrical recapture is plausible before sizing the full solar array.

⚡ TEG — Thermoelectric Generator Branch

TEG devices use the Seebeck effect — a temperature differential across a semiconductor junction produces a voltage. No moving parts. Very reliable. Efficiency is low (typically 5–8% at practical temperature differentials). TEGs are used on space probes powered by radioisotope heat sources (RTGs on Voyager, Cassini, Curiosity). At compute-node temperature differentials (50–100 K between cold loop and ambient) the output is small. The system flags TEG branches as low-significance recovery if their electrical output is below 1% of total internal dissipation.

When it helps: scavenging small amounts of power from unavoidable temperature gradients. Not architecture-defining at this scale.

🔦 Laser / RF Export Branch

Directed energy export branches model the conversion of electrical power (derived from thermal branches or the main solar array) into laser or RF beams directed away from the spacecraft. This is the only thermal branch that can actually reduce total onboard heat burden — because if energy leaves as a photon beam it does not return as heat. The catch: the conversion process (laser or RF amplifier) is itself lossy, and those losses become heat onboard.

When it helps: long-range power beaming concepts, inter-satellite laser link power budgets, or directed energy applications where exporting energy off-vehicle is the design intent.

10. Tab 7 — Output & Packet

Tab 7 is where the system produces authoritative output. Fill in Report Identity (org name, contact, confidentiality notice) — this appears in the formatted report and the packet artifact. Then click Build Report.

What the runtime returns

  • Thermal Load Summary: device count, per-device power, compute total, non-compute total, total reject load. These are the numbers the radiator must handle.
  • Radiator Sizing: effective area, area with margin, target surface temp, emissivity used. The display also shows whether these are runtime-computed or preview estimates — look for the green runtime ✓ badge.
  • Extension 3A Results: topology validity, zone execution order, resistance chain bounds, BOL/EOL lifecycle sizing. Green means the zone graph is coherent and validated.
  • Risk Summary: maturity class, thermal cycling risk, corrosion risk, contamination risk, packaging stress. Derived from the material and device choices you made.
  • Transform Trace: a step-by-step log of everything the runtime computed. If something looks wrong, start here to understand why.

Readiness Check

The readiness check runs before the output cards populate. Blockers (red) mean the packet has a physical or schema error that prevents valid output. Resolve them before trusting results. Warnings (yellow) mean something is flagged but not blocking — usually a research-required assumption or a non-critical inconsistency.

11. Everything is editable — what that means

Every preset seeds a default state. No preset locks any field. You can change device TDP after selecting a preset. You can change radiator emissivity. You can change zone temperatures. You can add or remove zones, change working fluids, swap materials, adjust load states — any of it.

The rule: as long as your edits do not cause a blocker flag, the runtime will accept the packet and return authoritative output. The system is designed for exploration. Edit freely. Run often. Compare results.

The only things you should not change arbitrarily without understanding the implications:

  • Zone upstream/downstream refs — these define the topology graph. Wrong refs break topology validation.
  • Convergence tolerance and runaway multiplier — these control when the iterative solver stops. The spec requires runaway multiplier ≥ 2.0.
  • Extension mode fields — switching between topology_only and full in 3A changes which calculations run.

Common blocker causes:

  • Zone graph has a cycle (heat can't flow in a circle without explicit convergence declaration)
  • Background sink temp is 0 K exactly when 3A lifecycle is active (physical minimum is 2.7 K, the cosmic microwave background)
  • Runaway multiplier set below 2.0
  • Required fields left empty when runtime needs them for computation

13. Runtime authority vs preview

The system uses this hierarchy without exception:

  1. Runtime output — server-executed, authoritative, what you trust
  2. Preview values — browser-calculated, useful for steering while editing, not the answer
  3. Exploratory outputs — Extensions 2 and 4, intentionally non-authoritative, for concept comparison only

If the preview radiator area shown in Tab 5 while editing differs from the runtime radiator area shown in Tab 7 after running — the runtime number is correct. The preview is an approximation using simplified math and the current UI state. The runtime normalizes, applies defaults, resolves catalogs, and executes the full validated formula stack.
